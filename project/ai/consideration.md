# Consideration

LLM Wikiとして、この構成がどのようにToken消費を抑え、AIの作業精度を上げられるかを考察します。

## Tokenをどの程度節約できそうか

小規模プロジェクトでも、README、設計資料、機能仕様、規約、テスト方針を毎回すべて読むと、数千から一万Token程度を消費しやすくなります。

この構成では、依頼内容に応じて読む資料を絞ります。

例:

- ボタン配置変更: `design.md` + `skills/frontend.md` + `features/todo.md`
- Resolver修正: `features/todo.md` + `skills/graphql.md` + `skills/backend.md`
- E2E失敗調査: `state.md` + `skills/testing.md` + 関連機能

全資料を読む場合と比べると、初動で読むTokenはおおよそ30%から60%程度まで削減できる可能性があります。プロジェクトが大きくなり、機能別資料が増えるほど削減効果は大きくなります。

ただし、節約量は各ファイルの粒度に依存します。1ファイルが長くなりすぎると、結局README中心構成と同じ問題が起きます。

さらに、[operations.md](operations.md) のような運用ルールを置くと、調査で得た再利用可能な知見を次回以降の入口に戻せます。これにより、毎回同じ質問で同じ資料を再探索するTokenを減らせます。

## state.mdの役割は適切か

`state.md` は適切です。ただし、役割を限定する必要があります。

適している情報:

- 現在作業中の内容
- 進行中PR
- 未完了機能
- 既知の技術的負債
- 近い将来の修正予定

適していない情報:

- 恒久的な仕様
- 詳細な議事録
- 古い作業ログ
- 設計判断の長文説明

AIにとって `state.md` は、作業前の衝突確認とコンテキスト復元に役立ちます。一方で、ここに仕様を書きすぎると `features/` と矛盾しやすくなります。

結論として、`state.md` は「現在地を示す短い地図」として使うのがよいです。

一方で、履歴まで `state.md` に入れると肥大化します。更新履歴は [log.md](log.md) に分け、`state.md` は現在判断に必要な情報だけへ絞るのが適切です。

## navigator.mdは必要か

必要です。

AIはファイル名だけでもある程度判断できますが、依頼内容と読む資料の対応が明示されていると、探索Tokenを減らせます。

`navigator.md` が特に有効な場面:

- UI変更だがAPI影響もあるかもしれない
- 認証とTodoのように複数機能にまたがる
- テスト失敗の原因調査
- 初見のAIエージェントが作業する

ただし、`navigator.md` に詳細仕様を書き始めると肥大化します。役割は「読む順番の決定」に限定するべきです。

## skillsはどこまで細分化すべきか

最初は作業種別で分けるのがよいです。

推奨:

- `frontend.md`
- `backend.md`
- `graphql.md`
- `testing.md`

プロジェクトが大きくなったら追加を検討:

- `database.md`
- `security.md`
- `accessibility.md`
- `performance.md`
- `observability.md`

細分化しすぎると、AIがどれを読むべきか迷い、`navigator.md` の負担が増えます。1つのskillが長くなり、複数の作業で毎回読みすぎる状態になったら分割のタイミングです。

判断基準:

- その知識は複数機能で再利用されるか
- その作業をするとき毎回必要か
- 独立したチェックリストにできるか
- 他のskillと同時に読む必要が少ないか

## operations.mdは必要か

LLM Wikiとして運用するなら必要です。

`index.md` と `navigator.md` は「読む資料を選ぶ」ためのファイルです。一方で、`operations.md` は「得た知識をどこに戻すか」を決めるためのファイルです。

AIエージェントは質問に答えるだけなら、既存資料を読んで回答すれば終わります。しかし、価値ある回答をWikiに戻さない場合、次回も似た探索を繰り返します。`operations.md` で Ingest / Query / Lint の扱いを決めておくと、Wikiが一度きりの参照資料ではなく、継続的に育つ作業記憶になります。

ただし、何でも保存するとノイズが増えます。保存するのは、恒久仕様、再利用可能な作業知識、複数資料を横断した判断、今後の注意点に限定するべきです。

## log.mdとstate.mdの分離

`state.md` は現在状態、`log.md` は履歴です。

この分離はToken節約にも効きます。作業開始時に必要なのは通常 `state.md` だけで、過去の更新履歴を全部読む必要はありません。なぜその構成になったかを追うときだけ `log.md` を読めばよくなります。

ログを残す利点:

- AIがいつ何を更新したか追える
- Lint時に古い判断を見つけやすい
- 人間がWikiの成長を監査できる

ログを残す注意点:

- 仕様本文を書かない
- 会話全文を貼らない
- 現在状態の代わりに使わない

## LLM Wikiとして改善できそうな点

改善案:

- 各Markdown冒頭に `読む条件` と `読まなくてよい条件` を必ず置く
- `features/` に関連コードパスを追加する
- `state.md` に最終更新日と更新者を入れる
- `log.md` にAIによる更新履歴を追記する
- `operations.md` にIngest / Query / Lintの実行条件を明記する
- `navigator.md` に依頼文サンプルを増やす
- 機能ごとに「影響しやすい資料」を明記する
- テスト失敗時用の `debugging.md` を追加する
- 大規模化したら `ai/index.generated.md` を自動生成する

さらに発展させるなら、Markdown front matterを導入して、AIが機械的に読む資料を選べるようにできます。

例:

```yaml
---
scope: frontend
features: [todo]
read_when:
  - ui_change
  - component_change
---
```

このようなメタ情報があると、将来的にAIエージェントやCLIツールが依頼内容から必要ファイルを自動選択しやすくなります。

もう一段進めるなら、`features/` と `skills/` では収まりにくい横断テーマを `concepts/` として分けられます。

例:

- `concepts/context-routing.md`: 依頼から読む資料を絞る考え方
- `concepts/overdue-detection.md`: Todo期限切れ判定の横断仕様
- `concepts/auth-boundary.md`: 認証境界とデータ分離

ただし、小規模プロジェクトの初期段階では必須ではありません。横断知識が複数ファイルに散らばり始めてから追加するのがよいです。

## 限界

AI向けWikiは、知識の整理と再利用には有効です。しかし、人間が理解しなくてよいという意味ではありません。

特に次の領域は、人間の確認が必要です。

- セキュリティや権限
- DBやAPIの恒久仕様
- UX判断
- 未確定事項を確定仕様へ移す判断
- AIが推測で補った可能性がある記述

Wikiが育つほど、AIは過去の記述を前提に回答します。誤った記述が入ると、その後の判断にも影響します。そのため、Lintと人間レビューを運用に含めることが重要です。

## 結論

この構成は、小規模プロジェクトでもAI向けドキュメント設計の効果を検証しやすい形です。

重要なのは、情報量を増やすことではなく、AIが読まなくてよい情報を判断できることです。

`index.md` は入口、`navigator.md` は読む順番、`operations.md` はWikiを育てる運用、`state.md` は現在地、`log.md` は履歴、`features/` は機能仕様、`skills/` は作業知識として分けることで、Token消費を抑えながら必要なコンテキストを維持しやすくなります。
