# Consideration

LLM Wikiとして、この構成がどのようにToken消費を抑え、AIの作業精度を上げられるかを考察します。

## Tokenをどの程度節約できそうか

小規模プロジェクトでも、README、設計資料、機能仕様、規約、テスト方針を毎回すべて読むと、数千から一万Token程度を消費しやすくなります。

この構成では、依頼内容に応じて読む資料を絞ります。

例:

- ボタン配置変更: `design.md` + `skills/frontend.md` + `features/todo.md`
- Resolver修正: `features/todo.md` + `skills/graphql.md` + `skills/backend.md`
- E2E失敗調査: `state.md` + `skills/testing.md` + 関連機能

全資料を読む場合と比べると、初動で読むTokenはおおよそ30%から60%程度まで削減できる可能性があります。プロジェクトが大きくなり、機能別資料が増えるほど削減効果は大きくなります。

ただし、節約量は各ファイルの粒度に依存します。1ファイルが長くなりすぎると、結局README中心構成と同じ問題が起きます。

## state.mdの役割は適切か

`state.md` は適切です。ただし、役割を限定する必要があります。

適している情報:

- 現在作業中の内容
- 進行中PR
- 未完了機能
- 既知の技術的負債
- 近い将来の修正予定

適していない情報:

- 恒久的な仕様
- 詳細な議事録
- 古い作業ログ
- 設計判断の長文説明

AIにとって `state.md` は、作業前の衝突確認とコンテキスト復元に役立ちます。一方で、ここに仕様を書きすぎると `features/` と矛盾しやすくなります。

結論として、`state.md` は「現在地を示す短い地図」として使うのがよいです。

## navigator.mdは必要か

必要です。

AIはファイル名だけでもある程度判断できますが、依頼内容と読む資料の対応が明示されていると、探索Tokenを減らせます。

`navigator.md` が特に有効な場面:

- UI変更だがAPI影響もあるかもしれない
- 認証とTodoのように複数機能にまたがる
- テスト失敗の原因調査
- 初見のAIエージェントが作業する

ただし、`navigator.md` に詳細仕様を書き始めると肥大化します。役割は「読む順番の決定」に限定するべきです。

## skillsはどこまで細分化すべきか

最初は作業種別で分けるのがよいです。

推奨:

- `frontend.md`
- `backend.md`
- `graphql.md`
- `testing.md`

プロジェクトが大きくなったら追加を検討:

- `database.md`
- `security.md`
- `accessibility.md`
- `performance.md`
- `observability.md`

細分化しすぎると、AIがどれを読むべきか迷い、`navigator.md` の負担が増えます。1つのskillが長くなり、複数の作業で毎回読みすぎる状態になったら分割のタイミングです。

判断基準:

- その知識は複数機能で再利用されるか
- その作業をするとき毎回必要か
- 独立したチェックリストにできるか
- 他のskillと同時に読む必要が少ないか

## LLM Wikiとして改善できそうな点

改善案:

- 各Markdown冒頭に `読む条件` と `読まなくてよい条件` を必ず置く
- `features/` に関連コードパスを追加する
- `state.md` に最終更新日と更新者を入れる
- `navigator.md` に依頼文サンプルを増やす
- 機能ごとに「影響しやすい資料」を明記する
- テスト失敗時用の `debugging.md` を追加する
- 大規模化したら `ai/index.generated.md` を自動生成する

さらに発展させるなら、Markdown front matterを導入して、AIが機械的に読む資料を選べるようにできます。

例:

```yaml
---
scope: frontend
features: [todo]
read_when:
  - ui_change
  - component_change
---
```

このようなメタ情報があると、将来的にAIエージェントやCLIツールが依頼内容から必要ファイルを自動選択しやすくなります。

## 結論

この構成は、小規模プロジェクトでもAI向けドキュメント設計の効果を検証しやすい形です。

重要なのは、情報量を増やすことではなく、AIが読まなくてよい情報を判断できることです。

`index.md` は入口、`navigator.md` は読む順番、`state.md` は現在地、`features/` は機能仕様、`skills/` は作業知識として分けることで、Token消費を抑えながら必要なコンテキストを維持しやすくなります。
