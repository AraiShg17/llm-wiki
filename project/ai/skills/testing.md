# Skill: Testing

テスト追加、修正、失敗調査に関係する作業で読む資料です。

## 読む条件

読む:

- テストを書く
- テストが失敗している
- 仕様変更の検証範囲を決める
- CI失敗を調査する

読まなくてよい:

- ドキュメントだけの変更
- 実装に影響しないメモ更新

## テスト種別

| 種別 | 対象 | 目的 |
| --- | --- | --- |
| Unit | 関数、Service | 業務ルールを速く検証 |
| Component | React component | 表示とユーザー操作を検証 |
| Integration | Resolver、Repository | API契約とDB連携を検証 |
| E2E | ブラウザ操作 | 主要ユーザーフローを検証 |

## 優先順位

1. 変更した業務ルールのUnit test
2. API契約が変わる場合のIntegration test
3. UI挙動が変わる場合のComponent test
4. 主要フローが変わる場合のE2E

すべてをE2Eで検証しないでください。E2Eは遅く壊れやすいため、主要導線に限定します。

## Todo機能の観点

Unit:

- タイトル必須
- 初期priority
- 期限切れ判定
- 他ユーザーTodoを操作できない

Component:

- Todo作成フォーム
- 完了チェック
- 削除確認
- 空状態

E2E:

- ログインしてTodoを作成する
- Todoを完了する
- Todoを削除する

## Auth機能の観点

- 未ログインでTodo画面へアクセスするとログインへ誘導される
- ログイン失敗時にエラー表示される
- ログアウト後に保護ページへ戻れない
- 他ユーザーのTodoをAPIで取得できない

## Notification機能の観点

- 期限切れTodoから通知が作られる
- 同じTodoの未読通知が重複しない
- Todo完了時に関連通知が既読になる
- 既読済み通知は未読数に含まれない

## 失敗調査の順序

1. 失敗しているテスト名と期待値を確認する
2. 関連する `features/*.md` を読む
3. 仕様変更による期待値更新か、実装バグかを分ける
4. fixtureやmockが古くないか確認する
5. 必要最小限のテスト修正にする

## AIへの注意

- テストが通るように仕様を弱めないでください。
- 実装詳細に依存したテストを増やさないでください。
- 失敗テストを削除する場合は、削除理由を明確にしてください。
