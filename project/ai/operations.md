# Operations

AI向けWikiを保守するための運用ルールです。

このファイルは、仕様や実装の詳細ではなく、AIが得た知識をどこへ書き戻すか、どのタイミングで整合性を確認するかを定義します。

## 目的

AIが毎回ゼロから知識を探し直す状態を避けます。

作業で得た情報のうち、次回以降も役立つものは `ai/` 配下の適切なファイルへ反映します。一方で、一時的な会話内容、未検証の推測、再利用性の低い調査結果は保存しません。

## レイヤー

このサンプルでは、LLM Wikiの考え方をプロジェクトドキュメント向けに簡略化します。

| レイヤー | このサンプルでの対応 | 所有者 |
| --- | --- | --- |
| Source | 実装コード、Issue、PR、会話で確定した要件 | 人間と実装 |
| AI Wiki | `ai/` 配下のMarkdown | AIが提案、必要に応じて人間が確認 |
| Schema | `index.md`, `navigator.md`, `operations.md`, `coding_rules.md` | 人間の意図を優先してAIが保守 |

重要:

- Sourceの事実をAI Wikiより優先する
- AI Wikiに根拠不明の仕様を書かない
- Schema系ファイルは頻繁に増やさず、役割を明確に保つ

## Operation: Ingest

新しい仕様、設計判断、PR内容、障害調査結果などをAI Wikiへ取り込む操作です。

実行する条件:

- 新しい機能仕様が確定した
- 既存仕様が変更された
- 調査で再利用可能な知見が得られた
- テスト方針や作業手順が更新された

手順:

1. 根拠になるSourceを確認する
2. 恒久仕様か、一時状態か、作業知識かを分類する
3. 更新先を決める
4. 関連リンクを追加する
5. [log.md](log.md) に短く追記する

更新先の目安:

| 内容 | 更新先 |
| --- | --- |
| 現在作業中、未完了、技術的負債 | [state.md](state.md) |
| 機能仕様 | `features/*.md` |
| 作業方法、確認事項 | `skills/*.md` |
| 読む順番、判断ルート | [navigator.md](navigator.md) |
| 全体構成 | [architecture.md](architecture.md) |
| UI設計方針 | [design.md](design.md) |
| 規約 | [coding_rules.md](coding_rules.md) |

保存しないもの:

- その場限りのデバッグログ
- 未確認の推測
- 具体的な会話の全文
- すぐ古くなる細かい進捗

## Operation: Query

AI Wikiを使って質問に答え、その回答のうち再利用価値があるものを保存する操作です。

実行する条件:

- 質問への回答が今後も繰り返し使われそう
- 複数資料を横断して判断した
- 既存ドキュメントに不足が見つかった

手順:

1. [navigator.md](navigator.md) で読む資料を絞る
2. 必要な `features/` と `skills/` だけ読む
3. 回答する
4. 再利用価値がある場合だけ、該当ファイルに要点を追記する
5. 更新した場合は [log.md](log.md) に記録する

Query結果を保存する判断基準:

- Yes: 「Todo期限切れ判定はどこで行うべきか」のように設計判断として再利用できる
- Yes: 「通知テストで毎回確認する境界値」のようにテスト観点として残せる
- No: 「この1回のエラー原因はtypoだった」のように汎用性が低い
- No: まだ検証していない仮説

## Operation: Lint

AI Wikiの健康診断です。

実行する条件:

- 大きな仕様変更後
- `features/` や `skills/` を複数更新した後
- AIが読むべき資料を迷ったとき
- 定期的な棚卸し

確認項目:

- リンク切れがないか
- `index.md` の一覧に新規ファイルが載っているか
- `navigator.md` の読む順番が実態と合っているか
- `state.md` に古い進捗が残っていないか
- 同じ知識が複数ファイルに矛盾して書かれていないか
- `features/` と `skills/` の責務が混ざっていないか
- 長すぎるファイルを分割すべきか

Lintで見つけた問題は、可能ならその場で小さく修正します。大きな再編が必要な場合は [state.md](state.md) の技術的負債に記録します。

## 人間レビューが必要な変更

次はAIだけで確定しないでください。

- APIやDBの恒久仕様変更
- 認証、権限、セキュリティに関わる変更
- `navigator.md` の大幅な再設計
- `skills/` の分割、統合、削除
- 未確定事項を確定仕様へ移す変更

## AIへの注意

- Wikiは回答の墓場ではなく、次回の判断を軽くするための構造です。
- 追記よりも、既存ページとの整合性を優先してください。
- 知識を増やすほど価値が出る一方、未検証情報も蓄積しやすくなります。根拠のない断定は避けてください。
- 人間が読んで理解できる量に保つため、長文化したページは分割を検討してください。
